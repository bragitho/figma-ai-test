Formal Evaluation Report

AI-Assisted Figma-to-Code Workflow vs. Webflow

Prepared for: Internal review

Subject: Recommended platform and workflow direction for content websites and booking flows

Date: April 16, 2026

1. Executive Summary

This report evaluates two possible directions for building and maintaining the company’s content websites and booking-related user experiences:

  • a Webflow-centered approach
  • a code-first workflow using Figma, Figma MCP, and an AI coding agent such as Codex

Based on the operating requirements discussed, the recommended direction is the code-first Figma MCP + AI-assisted engineering model.

This recommendation is based on the following core factors:

  • booking flows are embedded directly in the main frontend
  • the organization already depends on a complex Contentful setup
  • localization is a core requirement
  • forms are workflow-critical and tied to business logic
  • content pages are often connected to product and location data
  • engineers own production changes
  • the company expects significant custom logic growth
  • reducing long-term lock-in is important
  • the desired future state is a shared design system in code

In summary, Webflow is a strong platform for visually managed marketing websites, but it is not the right primary foundation for a business-critical, product-connected frontend. A code-first approach supported by Figma MCP and AI provides better alignment with the company’s architecture, governance model, and long-term technical goals.

2. Purpose of This Report

The purpose of this report is to document the reasoning behind the decision to prefer an AI-assisted Figma-to-code workflow over Webflow as the company’s primary implementation path.

The goal is not to dismiss Webflow as a product. Rather, the goal is to determine which approach is the better fit for the company’s specific requirements, team structure, and long-term platform strategy.

3. Evaluation Context

The following operational context was established during the discussion:

  • The company builds content websites and booking flows for a multi-profession business.
  • Booking engines for hotels, activities, and products are embedded within the main frontend.
  • Contentful is already used in a complex and mature way.
  • Localization is a core need and currently handled fairly cleanly.
  • Marketing and product owners compose pages from approved modules rather than designing fully unrestricted layouts.
  • SEO and promotional pages are often connected to product and location data.
  • Forms are workflow-critical and tied to business rules.
  • Engineers are the owners of production changes.
  • A shared design system in code is a desired future state.
  • AI-generated output will always be reviewed by engineers, but the objective is to achieve high-quality, near-shippable drafts.

4. Options Considered

4.1 Option A: Webflow as the Primary Platform

Under this model, Webflow would act as the main platform for layout, styling, content presentation, page composition, and possibly portions of forms and integrations.

This approach is typically strongest when:

  • the website is primarily a publishing or marketing surface
  • non-engineers need strong direct control over layout and page creation
  • the content model is relatively standard
  • custom business logic is limited
  • time-to-publish is more important than architectural flexibility

4.2 Option B: Figma MCP + AI-Assisted Code Generation in a Code-First Architecture

Under this model, Figma remains the design source, Figma MCP and an AI engineering agent accelerate translation into code, and the existing frontend codebase remains the production runtime.

This approach is strongest when:

  • the frontend is tightly coupled to business logic
  • engineering owns the production system
  • content must coexist with dynamic product or booking functionality
  • a code-native design system is a strategic goal
  • long-term maintainability and flexibility matter more than visual publishing convenience

5. Findings

5.1 The Company’s Frontend Behaves as a Product Surface, Not Just a Marketing Site

The booking flows are embedded in the main frontend. This means the website is not only a publishing layer. It is also a business-critical application surface.

This is a decisive architectural factor. A platform designed primarily for visual site building is not the best central runtime for a system that must also support booking logic, product behavior, location-specific rules, and operational workflows.

A code-first architecture is the more appropriate foundation.

5.2 The Existing Contentful Setup Should Remain the Content Source of Truth

The company already has a complex Contentful setup, and localization within that system is relatively clean.

Adopting Webflow as the primary platform would create pressure to duplicate or partially recreate content structures, editorial workflows, and rendering assumptions in another system. This would increase the risk of model drift and operational inconsistency over time.

The stronger option is to preserve:

  • Contentful as the content and localization source of truth
  • the frontend codebase as the rendering source of truth
  • AI as an accelerator within that existing model

5.3 Localization Requires Architectural Control

Localization is a core requirement, not a secondary convenience. Even with only two languages, localization affects more than translated copy. It also affects:

  • routing
  • SEO behavior
  • editorial review
  • legal and operational content
  • product and booking messaging
  • locale-aware rendering decisions

Since the current localization structure is already functioning well in Contentful, it is strategically preferable to keep localization inside the code-first application model rather than introducing a second platform layer.

5.4 Workflow-Critical Forms Should Remain in the Application Layer

The company’s forms are directly tied to location logic and booking workflows. This makes them operational components rather than simple marketing forms.

These forms require full control over:

  • validation
  • branching logic
  • downstream integrations
  • business rules
  • error handling
  • analytics
  • testing

For this reason, they are best implemented and maintained in code. A site-builder-native approach is not an appropriate core strategy for forms of this type.

5.5 Approved-Module Composition Does Not Require a Visual Builder Runtime

Marketing and product owners need to compose pages from approved modules rather than design fully new page structures from scratch.

This is an important distinction. It means the real requirement is not unrestricted visual editing. The real requirement is safe assembly of approved building blocks.

That need is well served by:

  • reusable code-backed page sections
  • a structured CMS composition model
  • controlled styling and variation options
  • engineering-owned rendering behavior

In this context, Webflow does not provide a decisive functional advantage.

5.6 Engineering Ownership Favors a Code-Centered System

Engineers are intended to own production changes. That means the primary production system should optimize for maintainability, reviewability, consistency, extensibility, testing, and long-term architectural control.

These priorities point directly to a code-first platform model.

5.7 Expected Growth in Custom Logic Increases the Risk of Platform Friction

The company expects substantial custom logic growth over the next 12 months.

This is a common failure point for teams that adopt visual website platforms as a central runtime. Such platforms can feel efficient early on, but become restrictive as domain-specific requirements expand.

A code-first approach is more appropriate because it:

  • scales more cleanly with business complexity
  • avoids platform workarounds
  • supports deeper integration with booking and product logic
  • keeps implementation flexible as requirements evolve

5.8 A Shared Design System in Code Is Better Supported by AI-Assisted Figma-to-Code Workflows

The organization wants to move from a mixed component landscape toward a shared design system in code.

That objective aligns very well with a workflow where:

  • designers work in Figma
  • Figma MCP exposes design context
  • AI helps produce first-draft implementations
  • engineers review, normalize, and merge components into a shared code library

5.9 Product-Connected Landing Pages Are Better Served by Code Ownership

The company’s SEO and promotional pages are often connected to products. That means these pages are not merely static editorial surfaces. They often depend on product-aware and location-aware data.

When page content is closely tied to business context, a code-first rendering layer becomes more valuable because it can manage logic, state, personalization, and integration behavior consistently.

5.10 A Lower-Lock-In Strategy Supports Long-Term Flexibility

The company has clearly stated that lock-in is undesirable.

The AI-assisted Figma MCP + code direction improves implementation speed without surrendering ownership of:

  • the frontend
  • the component system
  • the CMS integration model
  • business logic
  • form behavior
  • hosting/runtime decisions

6. Where Webflow Still Has Value

This report does not conclude that Webflow has no value. On the contrary, Webflow remains a strong option for:

  • standalone marketing websites
  • brochure sites
  • short-lived campaign microsites
  • visually rich landing pages with limited business logic
  • situations where non-engineers must publish independently and rapidly

For isolated use cases of that kind, Webflow could still be appropriate. However, those benefits do not outweigh the architectural disadvantages of using it as the company’s main platform.

7. Recommended Future-State Architecture

The recommended operating model is as follows:

  • Figma serves as the design source
  • Figma MCP and an AI engineering agent such as Codex accelerate implementation
  • the existing frontend codebase remains the only production rendering runtime
  • Contentful remains the source of truth for content and localization
  • booking engines and backend services remain the source of truth for commercial and operational logic
  • a shared design system in code becomes the long-term UI foundation

Under this model:

  • marketers compose pages from approved modules in Contentful
  • designers design those modules in Figma
  • AI helps engineers turn those designs into code faster
  • engineers review and ship production changes
  • all critical business behavior remains under normal engineering control

8. Strategic Recommendation

The company should adopt and invest in a Figma MCP + AI-assisted coding workflow as an accelerator for a code-first frontend architecture.

The company should not adopt Webflow as the primary platform for content websites and booking flows.

If Webflow is used at all, it should be limited to narrow, low-complexity, low-coupling marketing scenarios where long-term architectural integration is not important.

9. Conclusion

The decision is not fundamentally about choosing the more modern or more impressive tool. It is about choosing the model that best fits the company’s real operating conditions.

Webflow is strongest when the problem is visual publishing.

The company’s real problem is broader and more complex. It involves content, localization, booking logic, operational forms, product-connected landing pages, engineering governance, and long-term system ownership.

For that reason, the AI-assisted Figma-to-code direction is the better choice.

It supports design velocity without weakening architecture. It improves delivery speed without sacrificing control. Most importantly, it aligns with the company’s current reality and its intended future state.
